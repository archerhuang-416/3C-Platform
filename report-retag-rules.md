# 报告导入重打标分类规则文档

> 适用场景：从外部"专家纪要/报告库"网站爬取报告，导入自有平台时按本规则重新打标分类。
> 版本：v1.2　|　最后更新：2026-06-26

---

## 1. 设计目标与核心原则

原网站打标不准的根因是"标题命中关键词即归类"，导致"标题讲 A、正文实为 B"的报告被错分。本规则的核心原则：

- **标题只作线索，正文做裁决。** 标题用于快速出候选，最终归类需正文支撑。
- **主分类 + 多标签。** 每篇报告归一个主分类用于导航归档，同时保留多个标签描述其涉及的其他平台/品牌/主题。
- **仅疑难件调 AI。** 能用规则确定的不读正文，只有多命中、零命中、标题与正文矛盾时才调用 AI，控制成本与耗时。
- **疑难件进人工复核队列，不随机分配。** 保证结果可复现、可审计。
- **全程留痕。** 记录命中规则、置信度、AI 理由与证据，供审计与持续优化。

---

## 2. 分类体系（多维度）

分类是多维度并存的，一篇报告可同时属于多个维度。建议结构为"一个主分类 + 多个标签"。

| 维度（板块） | 类目（示例，按实际维护） |
|---|---|
| 竞对平台 | 拼多多、阿里、字节 |
| 品牌 | 苹果、华为、小米、OPPO、VIVO、荣耀 |
| 即时零售平台 | 美团闪购、京东秒送、淘宝闪购 |
| 其他 | 不归属上述任一板块的报告（与三大板块平行的独立板块） |
| 主题标签 | AI、芯片、补贴、UE、市场份额、跨境、供应链、产能规划 …… |

**主分类**取信号最强的一个；其余涉及到的维度全部作为**标签**保留。"其他"是与竞对平台、品牌、即时零售平台**平行的板块**，承接无法归入三大板块的报告。

> **注：** "美团"即"美团闪购"——不在竞对平台单独设"美团"类目，标题/正文出现"美团"一律映射到即时零售平台的 **美团闪购**。

## 3. 分类词典（最重要的基础资产）

所有"命中"判断都基于这份词典，而非字面匹配。每个类目维护：标准名、别名/同义词、关联实体。

### 3.1 词典模板（YAML 示例）

```yaml
categories:
  - canonical: 阿里                 # 标准名（= 主分类显示名）
    dimension: 竞对平台             # 所属板块
    aliases: [阿里巴巴, 淘宝, 天猫, 1688, 阿里妈妈,千问]
    entities: [阿里云, 飞猪, 菜鸟, 钉钉, 盒马,千问]   # 关联实体/子业务

  - canonical: 字节
    dimension: 竞对平台
    aliases: [字节跳动, 抖音, 抖音电商, 抖店, ByteDance, TikTok,豆包]
    entities: [今日头条, 巨量引擎, 抖音商城,豆包]

  - canonical: 拼多多
    dimension: 竞对平台
    aliases: [PDD, 多多, Temu, 多多买菜]
    entities: [多多视频]

  - canonical: 美团闪购             # "美团"即等于本类目
    dimension: 即时零售平台
    aliases: [美团, Meituan, 美团即时零售, 美团外卖, 美团买菜, 美团优选, 大众点评]
    entities: []

  - canonical: 淘宝闪购
    dimension: 即时零售平台
    aliases: [淘宝即时零售]
    entities: []

  - canonical: 华为
    dimension: 品牌
    aliases: [Huawei]
    entities: [昇腾, 鸿蒙, 海思, 麒麟]

# …其余类目同结构维护
```

### 3.2 维护要点

- **"美团"统一映射到"美团闪购"。** 词典里把"美团/美团外卖/大众点评"等都归到美团闪购类，不再单设竞对平台"美团"。
- **别名要覆盖口语、英文、子品牌。** 标题写"淘宝/抖音"应能分别匹配到"阿里/字节"类目。
- **关联实体用于正文命中加分。** 正文出现"昇腾/鸿蒙"可增强"华为"的信号。
- 词典是活文档，由人工复核的修正持续回流补充。

## 4. 处理流水线

### 第 0 步 · 归一化
清洗标题与正文，统一全半角、繁简、英文大小写，准备好可被词典扫描的文本。

### 第 1 步 · 标题出候选
用词典扫标题，得到命中的候选类目集合 `C`。注意：候选 ≠ 定案。

### 第 2 步 · 按分支裁决

```
              ┌─ |C| == 1 ──→ 分支 A
扫标题得到 C ─┼─ |C| >= 2 ──→ 分支 B
              └─ |C| == 0 ──→ 分支 D
```

**分支 A：标题恰好命中一个**
正文校验该词：是否高频、是否出现在摘要/首段。
- 正文确认 → 该词为主分类（**不调 AI**）。
- 正文里仅一笔带过、主题明显是别的 → 转分支 C。

**分支 B：标题命中两个及以上**
取专家"公司/任职"过一遍词典，与候选 `C` 比对：
- 恰好命中一个候选 → 该候选为主分类，其余候选降为标签（**不调 AI**）。
- 命中零个或多个 → 转分支 C。

**分支 C：调 AI 读正文（疑难件）**
让 AI 对每个候选打"中心度"分（0–1，"这篇主要在讲谁"，而非"有没有提到"），并返回一句判断理由。
- 最高分明显领先且 ≥ 阈值 → 它为主分类，其余高分项为标签。
- 多项接近、均 < 阈值 → **进人工复核队列**，标记"低置信"，附 AI 理由。

**分支 D：标题零命中**
交 AI 按正文主题归类；归不进现有类目 → 落"**其他**"板块。

### 贯穿规则 · 跨平台报告
天生横跨多平台的竞争/行业分析（如"云厂商算力…中外竞争格局"），主分类给"**其他**"板块，再挂多个标签，不硬塞进单一平台。

### 流程图

```
标题归一化
   │
   ├─ A 命中1个 ─→ 正文校验 ─┬─ 确认 ─→ 定主分类 ✅
   │                          └─ 矛盾 ─→ 分支C
   ├─ B 命中≥2 ─→ 专家身份比对 ─┬─ 命中1个 ─→ 定主分类+标签 ✅
   │                            └─ 0个/多个 ─→ 分支C
   ├─ C 调AI中心度打分 ─┬─ 最高分领先且过阈值 ─→ 定主分类+标签 ✅
   │                    └─ 接近/低于阈值 ─→ 人工复核队列 🔍
   └─ D 命中0个 ─→ AI主题归类 ─┬─ 命中类目 ─→ 定主分类 ✅
                                └─ 无 ─→ 其他板块 ✅
```

## 5. 留痕字段表

每篇报告处理后，记录以下字段，供审计、复现与持续优化。

| 字段 | 类型 | 说明 |
|---|---|---|
| `report_id` | string | 报告唯一 ID（去重用） |
| `title` | string | 报告标题 |
| `source_url` | string | 原始链接 |
| `crawled_at` | datetime | 爬取时间 |
| `original_tags` | array | 原网站的标签（保留备查） |
| `expert_identity` | string | 专家公司/任职（缺失则空） |
| `final_primary` | string | 最终主分类 |
| `final_tags` | array | 最终标签集合 |
| `matched_rule` | enum | 命中分支：A / B / C / D / 跨平台 |
| `candidates` | array | 标题命中的候选类目集合 |
| `ai_used` | bool | 是否调用了 AI |
| `ai_scores` | object | 各候选的中心度分（仅分支 C/D） |
| `ai_reason` | string | AI 一句话判断理由 |
| `confidence` | float | 最终置信度 0–1 |
| `review_status` | enum | auto / pending_review / reviewed |
| `reviewer` | string | 复核人（如有） |
| `reviewed_primary` | string | 人工修正后的主分类（如有） |

## 6. 阈值与参数（初始建议，待验证集校准）

| 参数 | 初始值 | 含义 |
|---|---|---|
| 正文高频阈值 | 关键词在正文出现 ≥ 3 次 | 分支 A 正文校验 |
| 中心度通过阈值 | 最高分 ≥ 0.6 | 分支 C 是否可自动定案 |
| 领先差阈值 | 最高分 − 次高分 ≥ 0.2 | 分支 C 是否"明显领先" |
| 低置信下限 | 最高分 < 0.6 | 触发人工复核 |

> 以上为初始值，必须用验证集回测后调整，切勿凭感觉定死。

## 7. 验证与持续优化

**验证集：** 先手工标注约 50 篇作为标准答案，覆盖各分支场景（单命中、多命中、零命中、跨平台）。跑完规则后计算主分类准确率与标签召回率，再调阈值。

**复核闭环：** 人工复核队列里的修正，定期回流：补充词典别名/关联实体、调整阈值。这是系统越用越准的核心来源。**主题标签（AI/补贴/UE…）同样纳入人工复核**——AI 在分支 C/D 顺带产出主题标签后，连同主分类一并进入复核范围，复核人可增删主题标签，修正同样回流词典。

**新类目建议机制：** 当落入"**其他**"板块的报告累积到阈值（建议：单一未知主题聚类 ≥ 10 篇，或"其他"占比超过 15%），系统自动提示运营评估是否**新增类目**，并附上该聚类的高频关键词与样例报告，辅助决策。

**配套工程：**
- **去重**：按 `report_id` 或标题+来源去重。
- **增量更新**：月度增量爬取，只处理新报告。
- **专家身份缺失**：匿名报告直接跳过分支 B 的身份判断，进分支 C。
- **可复现**：全程不使用随机分配，相同输入得相同输出。

## 8. 已确认决策（v1.2）

- **板块划分（四个平行板块）：** 竞对平台（拼多多/阿里/字节）、品牌（苹果/华为/小米/OPPO/VIVO/荣耀）、即时零售平台（美团闪购/京东秒送/淘宝闪购）、其他。
- **抖音并入"字节"：** 抖音/抖音电商等映射到竞对平台"字节"类目。
- **美团 = 美团闪购：** 不在竞对平台单设"美团"，所有"美团/外卖/大众点评"等统一映射到即时零售平台的"美团闪购"。
- **"其他"为平行板块：** 承接跨平台、零命中、无法归类的报告，不再使用"行业综合"命名。
- **主题标签走人工复核：** 主题标签与主分类一并进入人工复核范围，可增删修正。
- **"其他"触发新类目建议：** 落入"其他"累积达阈值时，自动提示运营评估新增类目。

### 后续可扩展
- 新类目建议的触发阈值（聚类篇数、占比）可按实际数据量微调。
